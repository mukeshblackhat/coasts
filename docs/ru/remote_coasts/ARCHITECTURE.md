# Архитектура

Удалённый coast разделяет выполнение между вашей локальной машиной и удалённым сервером. Опыт разработки не меняется, потому что демон прозрачно маршрутизирует каждую операцию через SSH-туннель.

## Разделение на два контейнера

Каждый удалённый coast создаёт два контейнера:

### Shell Coast (локальный)

Лёгкий Docker-контейнер на вашей машине. У него те же bind mounts, что и у обычного coast (`/host-project`, `/workspace`), но нет внутреннего Docker-демона и нет compose-сервисов. Его entrypoint — `sleep infinity`.

Shell coast существует по одной причине: он сохраняет [мост файловой системы](../concepts_and_terminology/FILESYSTEM.md), чтобы агенты и редакторы на стороне хоста могли редактировать файлы в `/workspace`. Эти изменения синхронизируются с удалённой машиной через [rsync and mutagen](FILE_SYNC.md).

### Remote Coast (удалённый)

Управляется `coast-service` на удалённой машине. Именно здесь происходит фактическая работа: полный DinD-контейнер, в котором запущены ваши compose-сервисы, с динамическими портами, выделенными для каждого сервиса.

## Слой SSH-туннелей

Демон связывает локальную и удалённую стороны с помощью двух типов SSH-туннелей:

### Прямые туннели (с локальной машины на удалённую)

Для каждого сервисного порта демон создаёт туннель `ssh -L`, который сопоставляет локальный динамический порт с соответствующим удалённым динамическим портом. Именно это позволяет `localhost:{dynamic_port}` достигать удалённого сервиса.

```text
ssh -N -L {local_dynamic}:localhost:{remote_dynamic} user@remote
```

Когда вы запускаете `coast ports`, столбец dynamic показывает эти локальные конечные точки туннелей.

### Обратные туннели (с удалённой машины на локальную)

[Shared services](../concepts_and_terminology/SHARED_SERVICES.md) (Postgres, Redis и т. д.) работают на вашей локальной машине. Демон создаёт туннели `ssh -R`, чтобы удалённый DinD-контейнер мог к ним обращаться:

```text
ssh -N -R 0.0.0.0:{remote_port}:localhost:{local_port} user@remote
```

Внутри удалённого DinD-контейнера сервисы подключаются к shared services через `host.docker.internal:{port}`, который разрешается в шлюз Docker bridge, где прослушивает обратный туннель.

На удалённом хосте в sshd должен быть включён параметр `GatewayPorts clientspecified`, чтобы обратные туннели привязывались к `0.0.0.0`, а не к `127.0.0.1`.

### Восстановление туннелей

SSH-туннели могут рваться, когда ваш ноутбук уходит в сон или меняется сеть. Демон запускает фоновый цикл проверки состояния, который:

1. Проверяет каждый динамический порт каждые 5 секунд через TCP connect.
2. Если все порты экземпляра недоступны, завершает устаревшие процессы туннелей для этого экземпляра и устанавливает их заново.
3. Если недоступна только часть портов (частичный сбой), заново устанавливает только отсутствующие туннели, не затрагивая исправные.
4. Очищает устаревшие привязки удалённых портов через `fuser -k` перед созданием новых обратных туннелей.

Восстановление выполняется для каждого экземпляра отдельно -- восстановление туннелей одного экземпляра никогда не нарушает работу другого.

## Цепочка проброса портов

Все порты в промежуточном слое являются динамическими. Канонические порты существуют только на конечных точках: внутри DinD-контейнера, где слушают сервисы, и на вашем localhost через [`coast checkout`](../concepts_and_terminology/CHECKOUT.md).

```text
localhost:3000 (canonical, via coast checkout / socat)
       ↓
localhost:{local_dynamic} (allocated by daemon port manager)
       ↓ SSH -L tunnel
remote:{remote_dynamic} (allocated by coast-service port manager)
       ↓ Docker port publish
DinD container :3000 (canonical, where the app listens)
```

Эта цепочка из трёх переходов позволяет запускать несколько экземпляров одного и того же проекта на одной удалённой машине без конфликтов портов. Каждый экземпляр получает собственный набор динамических портов с обеих сторон.

## Маршрутизация запросов

Каждый обработчик демона проверяет `remote_host` у экземпляра. Если он задан, запрос перенаправляется в coast-service через SSH-туннель:

| Command | Remote behavior |
|---------|-----------------|
| `coast run` | Создать shell coast локально + передать артефакты + перенаправить в coast-service |
| `coast build` | Сборка на удалённой машине (без пересылки локальной сборки) |
| `coast assign` | Rsync нового содержимого worktree + перенаправление запроса assign |
| `coast exec` | Перенаправление в coast-service |
| `coast ps` | Перенаправление в coast-service |
| `coast logs` | Перенаправление в coast-service |
| `coast stop` | Перенаправление + завершение локальных SSH-туннелей |
| `coast start` | Перенаправление + повторная установка SSH-туннелей |
| `coast rm` | Перенаправление + завершение туннелей + удаление локального shadow instance |
| `coast checkout` | Только локально (socat на хосте, перенаправление не требуется) |
| `coast secret set` | Сохранить локально + перенаправить в удалённый keystore |

## coast-service

`coast-service` — это плоскость управления, работающая на удалённой машине. Это HTTP-сервер (Axum), слушающий порт 31420, который отражает локальные операции демона: build, run, assign, exec, ps, logs, stop, start, rm, secrets и перезапуски сервисов.

Он управляет собственной базой данных состояния SQLite, Docker-контейнерами (DinD), выделением динамических портов, артефактами сборки, кэшем образов и зашифрованным keystore. Демон взаимодействует с ним исключительно через SSH-туннель -- coast-service никогда не выставляется в публичный интернет.

Инструкции по развёртыванию см. в [Setup](SETUP.md).
