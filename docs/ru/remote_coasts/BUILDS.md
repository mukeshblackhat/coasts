# Удалённые сборки

Удалённые сборки выполняются на удалённой машине через coast-service. Это гарантирует, что сборка использует собственную архитектуру удалённой машины (например, x86_64 на инстансе EC2) независимо от вашей локальной архитектуры (например, ARM Mac). Ни кросс-компиляция, ни эмуляция архитектуры не требуются.

## Как это работает

Когда вы запускаете `coast build --type remote`, происходит следующее:

1. Демон синхронизирует исходные файлы проекта (Coastfile, compose.yml, Dockerfiles, inject/) в удалённое рабочее пространство через SSH с помощью rsync.
2. Демон вызывает `POST /build` в coast-service через SSH-туннель.
3. coast-service выполняет полную сборку нативно на удалённой машине: `docker build`, загрузку образов, кэширование образов и извлечение секретов — всё в каталоге `/data/images/`.
4. coast-service возвращает `BuildResponse` с путём к артефакту и метаданными сборки.
5. Демон синхронизирует полный каталог артефакта (coastfile.toml, compose.yml, manifest.json, secrets/, inject/, tarball-файлы образов) обратно в `~/.coast/images/{project}/{build_id}/` на вашей локальной машине с помощью rsync.
6. Демон создаёт символическую ссылку `latest-remote`, указывающую на новую сборку.

```text
Local Machine                              Remote Machine
┌─────────────────────────────┐            ┌───────────────────────────┐
│  ~/.coast/images/my-app/    │            │  /data/images/my-app/     │
│    latest-remote -> {id}    │  ◀─rsync─  │    {id}/                  │
│    {id}/                    │            │      manifest.json        │
│      manifest.json          │            │      coastfile.toml       │
│      coastfile.toml         │            │      compose.yml          │
│      compose.yml            │            │      *.tar (images)       │
│      *.tar (images)         │            │                           │
└─────────────────────────────┘            └───────────────────────────┘
```

## Команды

```bash
# Build on the default remote (auto-selected if only one registered)
coast build --type remote

# Build on a specific remote
coast build --type remote --remote my-vm

# Build without running (standalone)
coast build --type remote
```

`coast run --type remote` также запускает сборку, если совместимой сборки ещё не существует.

## Соответствие архитектур

В `manifest.json` каждой сборки записывается архитектура, для которой она была собрана (например, `aarch64`, `x86_64`). Когда вы выполняете `coast run --type remote`, демон проверяет, соответствует ли существующая сборка архитектуре целевой удалённой машины:

- **Архитектура совпадает**: сборка переиспользуется. Пересборка не требуется.
- **Архитектура не совпадает**: демон ищет самую новую сборку с правильной архитектурой. Если такой не существует, он возвращает ошибку с рекомендацией выполнить пересборку.

Это означает, что вы можете один раз собрать проект на удалённой машине x86_64 и развёртывать его на любом количестве удалённых машин x86_64 без пересборки. Но вы не можете использовать ARM-сборку на удалённой машине x86_64 или наоборот.

## Символические ссылки

Удалённые сборки используют отдельную символическую ссылку от локальных сборок:

| Symlink | Points to |
|---------|-----------|
| `latest` | Самая недавняя локальная сборка |
| `latest-remote` | Самая недавняя удалённая сборка |
| `latest-{type}` | Самая недавняя локальная сборка определённого типа Coastfile |

Это разделение предотвращает ситуацию, при которой удалённая сборка перезаписывает вашу локальную символическую ссылку `latest`, и наоборот.

## Автоочистка

Coast хранит до 5 удалённых сборок для каждой пары `(coastfile_type, architecture)`. После каждой успешной удалённой сборки более старые сборки сверх этого лимита автоматически удаляются.

Сборки, которые используются запущенными инстансами, никогда не удаляются, независимо от лимита. Если у вас есть 7 удалённых сборок x86_64, но 3 из них используются активными инстансами, все эти 3 защищены.

Очистка учитывает архитектуру: если у вас есть удалённые сборки и `aarch64`, и `x86_64`, каждая архитектура независимо поддерживает собственный пул из 5 сборок.

## Хранение артефактов

Артефакты удалённой сборки хранятся в двух местах:

| Location | Path | Purpose |
|----------|------|---------|
| Remote | `/data/images/{project}/{build_id}/` | Источник истины на удалённой машине |
| Local | `~/.coast/images/{project}/{build_id}/` | Локальный кэш для повторного использования на разных удалённых машинах |

Кэш образов в `/data/image-cache/` на удалённой машине является общим для всех проектов, так же как `~/.coast/image-cache/` локально.

## Связь с локальными сборками

Удалённые и локальные сборки независимы друг от друга. `coast build` (без `--type remote`) всегда выполняет сборку на вашей локальной машине и обновляет символическую ссылку `latest`. `coast build --type remote` всегда выполняет сборку на удалённой машине и обновляет символическую ссылку `latest-remote`.

У вас могут одновременно существовать и локальные, и удалённые сборки одного и того же проекта. Локальные coast'ы используют локальные сборки; удалённые coast'ы используют удалённые сборки.

Подробнее о том, как работают сборки в целом (структура manifest, кэширование образов, типизированные сборки), см. в [Builds](../concepts_and_terminology/BUILDS.md).
