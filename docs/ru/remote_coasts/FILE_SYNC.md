# Синхронизация файлов

Удалённые coast используют двухуровневую стратегию синхронизации: rsync для массовой передачи, mutagen для непрерывной синхронизации в реальном времени. Оба инструмента являются зависимостями времени выполнения и устанавливаются внутри контейнеров coast — они не требуются на вашей хост-машине.

## Где выполняется синхронизация

```text
Local Machine                          Remote Machine
┌─────────────────────────────┐        ┌──────────────────────────────┐
│  coastd daemon              │        │                              │
│    │                        │        │                              │
│    │ rsync (direct SSH)     │  SSH   │  /data/workspaces/{p}/{i}/   │
│    │────────────────────────│───────▶│    (rsync writes here)       │
│    │                        │        │    │                         │
│    │ docker exec            │        │    │ bind mount              │
│    ▼                        │        │    ▼                         │
│  Shell Container            │  SSH   │  Remote DinD Container       │
│    /workspace (bind mount)  │───────▶│    /workspace                │
│    mutagen (continuous sync)│        │    (compose services running)│
│    SSH key (copied in)      │        │                              │
└─────────────────────────────┘        └──────────────────────────────┘
```

Демон запускает rsync напрямую из хост-процесса. Mutagen запускается внутри локального shell-контейнера через `docker exec`.

## Уровень 1: rsync (массовая передача)

При `coast run` и `coast assign` демон запускает rsync с хоста для передачи файлов рабочего пространства на удалённую машину:

```bash
rsync -rlDzP --delete-after \
  --rsync-path="sudo rsync" \
  --exclude '.git' --exclude 'node_modules' \
  --exclude 'target' --exclude '__pycache__' \
  --exclude '.react-router' --exclude '.next' \
  -e "ssh -p {port} -i {key}" \
  {local_workspace}/ {user}@{host}:{remote_workspace}/
```

После завершения rsync демон запускает `sudo chown -R` на удалённой машине, чтобы передать SSH-пользователю владение файлами. rsync запускается от root через `--rsync-path="sudo rsync"`, потому что удалённое рабочее пространство может содержать файлы, принадлежащие root, после операций coast-service внутри контейнера.

### Что rsync делает хорошо

- **Первоначальные передачи.** Первый `coast run` отправляет всё рабочее пространство.
- **Переключения worktree.** `coast assign` отправляет только дельту между старым и новым worktree. Файлы, которые не изменились, не передаются повторно.
- **Сжатие.** Флаг `-z` сжимает данные при передаче.

### Исключённые пути

rsync пропускает пути, которые не должны передаваться:

| Path | Why |
|------|-----|
| `.git` | Большой, не нужен на удалённой машине (достаточно содержимого worktree) |
| `node_modules` | Пересобирается внутри DinD из lockfile-ов |
| `target` | Артефакты сборки Rust/Go, пересобираются на удалённой машине |
| `__pycache__` | Кэш байткода Python, регенерируется |
| `.react-router` | Сгенерированные типы, создаются заново dev-сервером |
| `.next` | Кэш сборки Next.js, регенерируется |

### Защита сгенерированных файлов

Когда `coast assign` запускается с `--delete-after`, rsync обычно удаляет на удалённой машине файлы, которых нет локально. Это уничтожило бы сгенерированные файлы (например, proto-клиенты в `generated/`), которые были созданы удалённым dev-сервером, но отсутствуют в вашем локальном worktree.

Чтобы предотвратить это, rsync использует правила `--filter 'P generated/***'`, которые защищают определённые сгенерированные каталоги от удаления. Защищённые пути включают `generated/`, `.react-router/`, `internal/generated/` и `app/generated/`.

### Обработка частичной передачи

Код завершения rsync 23 (частичная передача) рассматривается как нефатальное предупреждение. Это обрабатывает состояние гонки, при котором работающие dev-серверы внутри удалённого DinD регенерируют файлы (например, `.react-router/types/`) в то время, когда rsync выполняет запись. Исходные файлы передаются успешно; не удаться могут только сгенерированные артефакты, а dev-сервер всё равно их пересоздаст.

## Уровень 2: mutagen (непрерывная синхронизация)

После первоначального rsync демон запускает сессию mutagen внутри локального shell-контейнера:

```bash
docker exec {shell_container} mutagen sync create \
    --name coast-{project}-{instance} \
    --sync-mode one-way-safe \
    --ignore-vcs \
    --ignore node_modules --ignore target \
    --ignore __pycache__ --ignore .next \
    /workspace/ {user}@{host}:{remote_workspace}/
```

Mutagen отслеживает изменения файлов через события уровня ОС (inotify внутри контейнера), группирует изменения и передаёт дельты по постоянному SSH-соединению. Ваши правки появляются на удалённой машине в течение нескольких секунд.

### Режим one-way-safe

Mutagen работает в режиме `one-way-safe`: изменения идут только с локальной машины на удалённую. Файлы, созданные на удалённой машине (dev-серверами, инструментами сборки и т. д.), не синхронизируются обратно на вашу локальную машину. Это предотвращает засорение рабочего каталога сгенерированными артефактами.

### Mutagen — зависимость времени выполнения

Mutagen установлен в:

- **образе coast** (собирается командой `coast build` из `[coast.setup]`), который используется локальным shell-контейнером.
- **Docker-образе coast-service** (`Dockerfile.coast-service`), который используется на удалённой стороне.

Демон никогда не запускает mutagen напрямую на хосте. Он управляет им через `docker exec` в shell-контейнер.

## Жизненный цикл

| Command | rsync | mutagen |
|---------|-------|---------|
| `coast run` | Первоначальная полная передача | Сессия создаётся после rsync |
| `coast assign` | Передача дельты нового worktree | Старая сессия завершается, создаётся новая |
| `coast stop` | -- | Сессия завершается |
| `coast rm` | -- | Сессия завершается |

### Поведение при отказе

Если сессию mutagen не удаётся запустить внутри shell-контейнера, демон записывает предупреждение в лог. Первоначальный rsync всё равно предоставляет содержимое рабочего пространства, но изменения файлов не будут синхронизироваться в реальном времени, пока сессия не будет восстановлена (например, при следующем `coast assign` или перезапуске демона).

## Конфигурация стратегии синхронизации

Раздел `[remote]` в вашем Coastfile управляет стратегией синхронизации:

```toml
[remote]
workspace_sync = "mutagen"    # "rsync" (по умолчанию) or "mutagen"
```

- **`rsync`** (по умолчанию): выполняется только первоначальная передача rsync. Непрерывной синхронизации нет. Подходит для сред CI или пакетных задач, где синхронизация в реальном времени не нужна.
- **`mutagen`**: rsync для первоначальной передачи, затем mutagen для непрерывной синхронизации. Используйте это для интерактивной разработки, когда вы хотите, чтобы правки появлялись на удалённой машине немедленно.
