# Директории worktree

Поле `worktree_dir` в `[coast]` управляет тем, где размещаются git worktree. Coast использует git worktree, чтобы дать каждому инстансу собственную копию кодовой базы на другой ветке без дублирования всего репозитория.

## Синтаксис

`worktree_dir` принимает одну строку или массив строк:

```toml
# Single directory (default)
worktree_dir = ".worktrees"

# Multiple directories
worktree_dir = [".worktrees", ".claude/worktrees", "~/.codex/worktrees"]
```

Если поле не указано, по умолчанию используется `".worktrees"`.

## Типы путей

### Относительные пути

Пути, которые не начинаются с `~/` или `/`, разрешаются относительно корня проекта. Это наиболее распространённый вариант, и он не требует специальной обработки — такие пути находятся внутри директории проекта и автоматически доступны внутри контейнера Coast через стандартный bind mount `/host-project`.

```toml
worktree_dir = ".worktrees"
worktree_dir = [".worktrees", ".claude/worktrees"]
```

### Пути с тильдой (внешние)

Пути, начинающиеся с `~/`, разворачиваются в домашнюю директорию пользователя и рассматриваются как **внешние** директории worktree. Coast добавляет отдельный bind mount, чтобы контейнер мог получить к ним доступ.

```toml
worktree_dir = ["~/.codex/worktrees", ".worktrees"]
```

Так выполняется интеграция с инструментами, которые создают worktree вне корня вашего проекта, например OpenAI Codex (который всегда создаёт worktree в `$CODEX_HOME/worktrees`).

### Абсолютные пути (внешние)

Пути, начинающиеся с `/`, также считаются внешними и получают собственный bind mount.

```toml
worktree_dir = ["/shared/worktrees", ".worktrees"]
```

### Шаблоны glob (внешние)

Внешние пути могут содержать метасимволы glob (`*`, `?`, `[...]`).

```toml
worktree_dir = [".worktrees", "~/.shep/repos/*/wt"]
```

Это полезно, когда инструмент создаёт worktree по пути с компонентом, который различается для каждого проекта (например, хешем). `*` соответствует любому одному имени директории, поэтому `~/.shep/repos/*/wt` соответствует `~/.shep/repos/a21f0cda9ab9d456/wt` и любой другой директории с хешем, содержащей поддиректорию `wt`.

Поддерживаемый синтаксис glob:

- `*` — соответствует любой последовательности символов в пределах одного компонента пути
- `?` — соответствует любому одному символу
- `[abc]` — соответствует любому символу из набора
- `[!abc]` — соответствует любому символу, не входящему в набор

Coast монтирует **корень glob** — префикс директории до первого компонента с подстановочным символом — а не каждое отдельное совпадение. Для `~/.shep/repos/*/wt` корнем glob является `~/.shep/repos/`. Это означает, что новые директории, появляющиеся после создания контейнера (например, новая директория с хешем, созданная Shep), автоматически доступны внутри контейнера без пересоздания. Динамические назначения на worktree по новым glob-совпадениям начинают работать сразу.

Добавление *нового* шаблона glob в Coastfile всё ещё требует `coast run`, чтобы создать bind mount. Но как только шаблон существует, новые директории, соответствующие ему, подхватываются автоматически.

## Как работают внешние директории

Когда Coast обнаруживает внешнюю директорию worktree (путь с тильдой или абсолютный путь), происходят три вещи:

1. **Bind mount контейнера** — Во время создания контейнера (`coast run`) разрешённый путь на хосте монтируется в контейнер как bind mount в `/host-external-wt/{index}`, где `{index}` — это позиция в массиве `worktree_dir`. Это делает внешние файлы доступными внутри контейнера.

2. **Фильтрация по проекту** — Внешние директории могут содержать worktree для нескольких проектов. Coast использует `git worktree list --porcelain` (которая по своей природе ограничена текущим репозиторием), чтобы обнаружить только те worktree, которые принадлежат этому проекту. Git watcher также проверяет принадлежность, читая файл `.git` каждого worktree и проверяя, что его указатель `gitdir:` разрешается обратно в текущий репозиторий.

3. **Перемонтирование workspace** — Когда вы выполняете `coast assign` к внешнему worktree, Coast перемонтирует `/workspace` из пути внешнего bind mount вместо обычного `/host-project/{dir}/{name}`.

## Именование внешних worktree

Внешние worktree с checkout веткой отображаются по имени своей ветки, так же как и локальные worktree.

Внешние worktree в состоянии **detached HEAD** (обычно это встречается у Codex) отображаются с использованием их относительного пути внутри внешней директории. Например, worktree Codex по адресу `~/.codex/worktrees/a0db/coastguard-platform` отображается как `a0db/coastguard-platform` в UI и CLI.

## `default_worktree_dir`

Управляет тем, какая директория используется, когда Coast создаёт **новый** worktree (например, когда вы назначаете ветку, у которой ещё нет существующего worktree). По умолчанию используется первая запись в `worktree_dir`.

```toml
[coast]
name = "my-app"
worktree_dir = [".worktrees", "~/.codex/worktrees"]
default_worktree_dir = ".worktrees"
```

Внешние директории никогда не используются для создания новых worktree — Coast всегда создаёт worktree в локальной (относительной) директории. Поле `default_worktree_dir` нужно только в том случае, если вы хотите переопределить значение по умолчанию (первую запись).

## Примеры

### Интеграция с Codex

OpenAI Codex создаёт worktree по адресу `~/.codex/worktrees/{hash}/{project-name}`. Чтобы они были видимы и доступны для назначения в Coast:

```toml
[coast]
name = "my-app"
worktree_dir = [".worktrees", "~/.codex/worktrees"]
```

После этого worktree Codex появятся в модальном окне checkout и в выводе `coast ls`. Вы можете назначить инстанс Coast на worktree Codex, чтобы запускать его код в полноценной среде разработки.

Примечание: после добавления внешней директории контейнер необходимо пересоздать (`coast run`), чтобы bind mount вступил в силу. Простого перезапуска существующего инстанса недостаточно.

### Интеграция с Claude Code

Claude Code создаёт worktree внутри проекта в `.claude/worktrees/`. Поскольку это относительный путь (внутри корня проекта), он работает как любая другая локальная директория worktree — внешний mount не требуется:

```toml
[coast]
name = "my-app"
worktree_dir = [".worktrees", ".claude/worktrees"]
```

### Интеграция с Shep

Shep создаёт worktree по адресу `~/.shep/repos/{hash}/wt/{branch-slug}`, где хеш уникален для каждого репозитория. Используйте шаблон glob, чтобы сопоставить директорию с хешем:

```toml
[coast]
name = "my-app"
worktree_dir = [".worktrees", "~/.shep/repos/*/wt"]
```

### Все harness вместе

```toml
[coast]
name = "my-app"
worktree_dir = [".worktrees", ".claude/worktrees", "~/.codex/worktrees", "~/.shep/repos/*/wt"]
```

## Живое чтение Coastfile

Изменения `worktree_dir` в вашем Coastfile вступают в силу немедленно для **списка** worktree (API и git watcher читают актуальный Coastfile с диска, а не только кэшированный артефакт сборки). Однако внешние **bind mount** создаются только во время создания контейнера, поэтому для того, чтобы новая внешняя директория стала доступной для монтирования, необходимо пересоздать инстанс.
