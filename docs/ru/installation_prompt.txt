Вы устанавливаете Coasts в этот проект. Coast (контейнеризованный хост) — это CLI-инструмент, который запускает несколько изолированных сред разработки на одной машине, используя Docker-in-Docker контейнеры. Каждая среда получает собственные порты, тома и рантайм — идеально для параллельных workflow с worktree.

Ваша задача: проанализировать этот проект и сгенерировать Coastfile (TOML-файл с именем "Coastfile" в корне проекта).

=== ДОКУМЕНТАЦИЯ ===

У Coast есть встроенная документация, доступная из CLI. Используйте её, чтобы понять полную схему Coastfile, стратегии томов, поведение assign и другие параметры конфигурации перед генерацией Coastfile.

Просмотрите дерево документации:

  coast docs

Эта команда выводит полное дерево документации. Начните с чтения файлов README — они предоставляют индексы, которые помогут найти нужную документацию по каждой теме:

  coast docs --path README.md
  coast docs --path coastfiles/README.md
  coast docs --path concepts_and_terminology/README.md

Прочитайте конкретный документ:

  coast docs --path coastfiles/PROJECT.md
  coast docs --path coastfiles/VOLUMES.md

Поиск по документации (семантический поиск — опишите, что вы ищете, на естественном языке):

  coast search-docs "how do volume strategies work"
  coast search-docs "shared postgres across instances"
  coast search-docs "secret injection from environment variables"

Используйте документацию, чтобы принимать обоснованные решения о конфигурации Coastfile для этого проекта. Раздел coastfiles/ подробно покрывает каждую директиву Coastfile.

=== СХЕМА COASTFILE (краткая справка) ===

[coast] — Обязательно. Метаданные проекта.

  name          (string, required)  Идентификатор проекта, используемый в именовании контейнеров/томов.
  compose       (string, optional)  Путь к docker-compose.yml относительно Coastfile.
  runtime       (string, optional)  "dind" (по умолчанию), "sysbox" или "podman".
  root          (string, optional)  Переопределение корня проекта (относительный или абсолютный путь).
  worktree_dir  (string, optional)  Директория для git worktree (по умолчанию: ".worktrees"). Автоопределяется по существующим worktree во время выполнения.

[coast.setup] — Необязательно. Настройка самого DinD-контейнера.

  packages    (array of strings)  Пакеты Alpine для установки (например, ["nodejs", "npm", "git"]).
  run         (array of strings)  Произвольные команды для запуска во время настройки.

[ports] — Обязательно (как минимум один). Карта логического имени в номер порта.
  Эти порты пробрасываются на хост, когда coast checkout выполнен.

  Пример:
  [ports]
  web = 3000
  api = 8080
  postgres = 5432

[volumes.*] — Необязательно. Конфигурация по каждому тому.

  strategy        "isolated" (по умолчанию) или "shared"
  service         Имя сервиса из Compose, который владеет этим томом.
  mount           Путь монтирования внутри контейнера сервиса.
  snapshot_source (только isolated) Инициализировать из существующего имени тома.

  Пример:
  [volumes.postgres_data]
  strategy = "isolated"
  service = "db"
  mount = "/var/lib/postgresql/data"

[secrets.*] — Необязательно. Извлечение и инъекция секретов.

  extractor   "file", "env", "command" или "macos-keychain"
  inject      "env:VAR_NAME" или "file:/path/in/container"
  ttl         Необязательное время истечения (например, "1h", "30m").

  Параметры, специфичные для extractor:
    file:            path = "./path/to/secret"
    env:             var = "HOST_ENV_VAR"
    command:         run = "echo secret-value"
    macos-keychain:  item = "keychain-item-name"

  Пример:
  [secrets.db_password]
  extractor = "env"
  var = "DB_PASSWORD"
  inject = "env:DATABASE_PASSWORD"

[inject] — Необязательно. Инъекция не-секретных файлов/переменных окружения с хоста.

  env     Массив имён переменных окружения хоста для проброса.
  files   Массив путей к файлам на хосте для монтирования.

  Пример:
  [inject]
  env = ["NODE_ENV", "DEBUG"]
  files = ["~/.ssh/id_ed25519", "~/.gitconfig"]

[shared_services.*] — Необязательно. Сервисы на хостовом Docker daemon, общие для всех инстансов.

  image           Docker image.
  ports           Массив номеров портов.
  volumes         Массив монтирований томов.
  env             Inline-таблица переменных окружения.
  auto_create_db  (bool) Автоматически создать базу данных на инстанс.
  inject          Инъекция строки подключения в контейнеры coast.

  Пример:
  [shared_services.postgres]
  image = "postgres:16-alpine"
  ports = [5432]
  volumes = ["postgres_data:/var/lib/postgresql/data"]
  env = { POSTGRES_USER = "dev", POSTGRES_PASSWORD = "dev", POSTGRES_DB = "app" }
  auto_create_db = true
  inject = "env:DATABASE_URL"

[assign] — Необязательно. Управляет тем, что происходит при переключении ветки (coast assign).

  default                 "none", "restart" или "rebuild"
  [assign.services]       Переопределения по сервисам.
  [assign.rebuild_triggers]  Глоб-шаблоны файлов по сервисам, которые триггерят rebuild.

  Пример:
  [assign]
  default = "none"
  [assign.services]
  api = "restart"
  worker = "rebuild"
  [assign.rebuild_triggers]
  worker = ["Dockerfile", "package.json"]

[services.*] — Необязательно. Сервисы как «голые» процессы (docker-compose не нужен).

  command   Команда shell для запуска.
  port      Номер порта.
  restart   "on-failure" или "always".

  Пример:
  [services.web]
  command = "node server.js"
  port = 3000
  restart = "on-failure"

=== ПРИМЕР: Минимальный (без compose) ===

[coast]
name = "my-app"
runtime = "dind"

[coast.setup]
packages = ["nodejs", "npm"]

[ports]
app = 3000

=== ПРИМЕР: С docker-compose ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000
postgres = 5432
redis = 6379

[volumes.postgres_data]
strategy = "shared"
service = "db"
mount = "/var/lib/postgresql/data"

[volumes.redis_data]
strategy = "isolated"
service = "cache"
mount = "/data"

[assign]
default = "none"

[assign.services]
app = "rebuild"

=== ПРИМЕР: С секретами ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000

[secrets.api_key]
extractor = "env"
var = "API_KEY"
inject = "env:API_KEY"

[secrets.ssh_key]
extractor = "file"
path = "~/.ssh/id_ed25519"
inject = "file:/run/secrets/ssh_key"

=== КЛЮЧЕВЫЕ КОМПРОМИССЫ ДЛЯ ОБСУЖДЕНИЯ С ПОЛЬЗОВАТЕЛЕМ ===

Перед генерацией Coastfile спросите пользователя о любых неоднозначных вариантах конфигурации. Вот основные:

Стратегия базы данных и инфраструктуры — есть три варианта для сервисов вроде postgres и redis:
  - Изолированные тома (по умолчанию): каждый инстанс Coast получает собственную копию данных внутри своего DinD-контейнера. Инстансы не могут мешать друг другу. Лучше всего, когда вы хотите состояние базы данных на ветку.
  - Общие тома: все инстансы читают и пишут в один и тот же том внутри своих DinD-контейнеров. Экономит место на диске, но параллельные записи из нескольких инстансов могут повредить данные.
  - Общие сервисы: запускайте базу данных на хостовом Docker daemon, а не внутри каждого Coast. Все инстансы подключаются к одному общему серверу. Использует меньше всего памяти, поддерживает auto_create_db для баз данных на инстанс на одном postgres, а данные переживают удаление инстанса. Лучше всего для больших команд или машин с ограниченной памятью.
  - Если в проекте есть база данных, спросите пользователя, какой подход им нужен. Объясните компромиссы — isolated самый безопасный, shared services наиболее эффективен по памяти.

Стратегия assign — что происходит при переключении Coast между worktree:
  - "none": ничего не делать (для сервисов вроде postgres/redis, которые не меняются между ветками).
  - "restart": перезапустить контейнер (для интерпретируемых сервисов, которым нужен лишь перезапуск процесса).
  - "rebuild": пересобрать Docker image и перезапустить (для сервисов, где смена ветки влияет на Dockerfile или зависимости сборки).
  - Если в проекте несколько сервисов, спросите, какие из них нужно пересобирать, а какие — перезапускать при переключении ветки.

=== ИНСТРУКЦИИ ===

1. Посмотрите на структуру проекта. Если есть docker-compose.yml, прочитайте его, чтобы определить сервисы, порты и тома.
2. Определите существующую директорию git worktree. Запустите `git worktree list`, чтобы проверить, настроены ли в проекте git worktree.
   - Если worktree существуют, изучите их пути, чтобы определить общую родительскую директорию (например, если worktree находятся в `../.worktrees/feat-a` и `../.worktrees/feat-b`, то worktree_dir равен `"../.worktrees"`).
   - Установите `worktree_dir` в Coastfile в соответствии с обнаруженной директорией.
   - Если worktree отсутствуют, не указывайте `worktree_dir` (Coast по умолчанию использует ".worktrees"). НЕ используйте ".coasts" — это засоряет проект директориeй с брендингом Coast.
3. Прочитайте соответствующую документацию Coast (используйте `coast docs` и `coast search-docs`), чтобы понять стратегии томов, поведение assign и любые параметры конфигурации, применимые к стеку этого проекта.
4. Спросите пользователя о любых неоднозначных вариантах конфигурации (см. компромиссы выше). Не угадывайте — объясните варианты и дайте им выбрать.
5. Сгенерируйте Coastfile в корне проекта на основе анализа проекта и ввода пользователя.
6. Если в проекте нет docker-compose.yml, используйте [services.*] для определений «голых» процессов или [coast.setup] для установки зависимостей.
7. Запустите `coast build`. Если не удаётся, проверьте ошибку и обратитесь к документации (`coast search-docs "<error description>"`) для устранения неполадок.
8. Запустите `coast run dev-1`. Если не удаётся, проверьте ошибку и обратитесь к документации.
9. Запустите `coast ui`, чтобы открыть панель Coastguard (это для пользователя, когда вы закончите).
