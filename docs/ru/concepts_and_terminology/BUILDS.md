# Сборки

Думайте о сборке coast как об образе Docker с дополнительной помощью. Сборка — это артефакт на основе каталога, который объединяет всё необходимое для создания экземпляров Coast: разрешённый [Coastfile](COASTFILE_TYPES.md), переписанный compose-файл, заранее загруженные tarball-архивы OCI-образов и внедрённые файлы хоста. Это не сам Docker-образ, но он содержит Docker-образы (в виде tarball-архивов) плюс метаданные, которые нужны Coast, чтобы связать их вместе.

## Что делает `coast build`

Когда вы запускаете `coast build`, демон выполняет следующие шаги по порядку:

1. Разбирает и проверяет Coastfile.
2. Читает compose-файл и отфильтровывает пропущенные сервисы.
3. Извлекает [секреты](SECRETS.md) из настроенных экстракторов и сохраняет их в зашифрованном виде в хранилище ключей.
4. Собирает Docker-образы для compose-сервисов, у которых есть директивы `build:` (на хосте).
5. Загружает Docker-образы для compose-сервисов, у которых есть директивы `image:`.
6. Кэширует все образы как OCI tarball-архивы в `~/.coast/image-cache/`.
7. Если настроен `[coast.setup]`, собирает пользовательский базовый DinD-образ с указанными пакетами, командами и файлами.
8. Записывает каталог артефакта сборки с манифестом, разрешённым coastfile, переписанным compose-файлом и внедрёнными файлами.
9. Обновляет символическую ссылку `latest`, чтобы она указывала на новую сборку.
10. Автоматически удаляет старые сборки сверх лимита хранения.

## Где хранятся сборки

```text
~/.coast/
  images/
    my-project/
      latest -> a3c7d783_20260227143000       (symlink)
      a3c7d783_20260227143000/                (versioned build)
        manifest.json
        coastfile.toml
        compose.yml
        inject/
      b4d8e894_20260226120000/                (older build)
        ...
  image-cache/                                (shared tarball cache)
    postgres_16_a1b2c3d4e5f6.tar
    redis_7_f6e5d4c3b2a1.tar
    coast-built_my-project_web_latest_...tar
```

Каждая сборка получает уникальный **ID сборки** в формате `{coastfile_hash}_{YYYYMMDDHHMMSS}`. Хеш включает содержимое Coastfile и разрешённую конфигурацию, поэтому изменения в Coastfile приводят к новому ID сборки.

Символическая ссылка `latest` всегда указывает на самую свежую сборку для быстрого определения. Если ваш проект использует типизированные Coastfile (например, `Coastfile.light`), каждый тип получает собственную символическую ссылку: `latest-light`.

Кэш образов в `~/.coast/image-cache/` используется совместно всеми проектами. Если два проекта используют один и тот же образ Postgres, tarball-архив кэшируется один раз.

## Что содержит сборка

Каждый каталог сборки содержит:

- **`manifest.json`** -- полные метаданные сборки: имя проекта, временную метку сборки, хеш coastfile, список кэшированных/собранных образов, имена секретов, пропущенные сервисы, [стратегии томов](VOLUMES.md) и другое.
- **`coastfile.toml`** -- разрешённый Coastfile (объединённый с родительским при использовании `extends`).
- **`compose.yml`** -- переписанную версию вашего compose-файла, в которой директивы `build:` заменены тегами заранее собранных образов, а пропущенные сервисы удалены.
- **`inject/`** -- копии файлов хоста из `[inject].files` (например, `~/.gitconfig`, `~/.npmrc`).

## Сборки не содержат секретов

Секреты извлекаются на этапе сборки, но хранятся в отдельном зашифрованном хранилище ключей по адресу `~/.coast/keystore.db` -- а не внутри каталога артефакта сборки. В манифесте записываются только **имена** извлечённых секретов, но никогда не их значения.

Это означает, что артефакты сборки безопасно просматривать без раскрытия чувствительных данных. Секреты расшифровываются и внедряются позже, когда экземпляр Coast создаётся с помощью `coast run`.

## Сборки и Docker

Сборка включает три вида Docker-образов:

- **Собранные образы** -- compose-сервисы с директивами `build:` собираются на хосте через `docker build`, получают тег `coast-built/{project}/{service}:latest` и сохраняются как tarball-архивы в кэше образов.
- **Загруженные образы** -- compose-сервисы с директивами `image:` загружаются и сохраняются как tarball-архивы.
- **Образ Coast** -- если настроен `[coast.setup]`, пользовательский Docker-образ собирается поверх `docker:dind` с указанными пакетами, командами и файлами. Получает тег `coast-image/{project}:{build_id}`.

Во время выполнения ([`coast run`](RUN.md)) эти tarball-архивы загружаются во внутренний [демон DinD](RUNTIMES_AND_SERVICES.md) через `docker load`. Именно это позволяет экземплярам Coast запускаться быстро без необходимости загружать образы из реестра.

## Сборки и экземпляры

Когда вы запускаете [`coast run`](RUN.md), Coast определяет последнюю сборку (или конкретный `--build-id`) и использует её артефакты для создания экземпляра. ID сборки записывается в экземпляр.

Вам не нужно пересобирать проект для создания новых экземпляров. Одна сборка может обслуживать много экземпляров Coast, работающих параллельно.

## Когда пересобирать

Пересобирайте только тогда, когда меняются ваш Coastfile, `docker-compose.yml` или конфигурация инфраструктуры. Пересборка ресурсоёмка -- она заново загружает образы, заново собирает Docker-образы и заново извлекает секреты.

Изменения в коде не требуют пересборки. Coast монтирует каталог вашего проекта напрямую в каждый экземпляр, поэтому обновления кода подхватываются немедленно.

## Автоматическая очистка

Coast хранит до 5 сборок на каждый тип Coastfile. После каждого успешного `coast build` старые сборки сверх лимита автоматически удаляются.

Сборки, которые используются работающими экземплярами, никогда не удаляются, независимо от лимита. Если у вас 7 сборок, но 3 из них обслуживают активные экземпляры, все эти 3 защищены.

## Удаление вручную

Вы можете удалять сборки вручную через `coast rm-build` или через вкладку Builds в Coastguard.

- **Полное удаление проекта** (`coast rm-build <project>`) требует, чтобы все экземпляры были сначала остановлены и удалены. Оно удаляет весь каталог сборок, связанные Docker-образы, тома и контейнеры.
- **Выборочное удаление** (по ID сборки, доступно в интерфейсе Coastguard) пропускает сборки, которые используются работающими экземплярами.

## Типизированные сборки

Если ваш проект использует несколько Coastfile (например, `Coastfile` для конфигурации по умолчанию и `Coastfile.snap` для томов, инициализированных снимком), каждый тип поддерживает собственную символическую ссылку `latest-{type}` и собственный пул очистки из 5 сборок.

```bash
coast build              # uses Coastfile, updates "latest"
coast build --type snap  # uses Coastfile.snap, updates "latest-snap"
```

Очистка сборки `snap` никогда не затрагивает сборки `default`, и наоборот.

## Удалённые сборки

При сборке для [удалённого coast](REMOTES.md) сборка выполняется на удалённой машине через `coast-service`, чтобы образы использовали нативную архитектуру удалённой машины. Затем артефакт передаётся обратно на вашу локальную машину для повторного использования. Удалённые сборки поддерживают собственную символическую ссылку `latest-remote` и очищаются отдельно для каждой архитектуры. Подробности см. в разделе [Remotes](REMOTES.md).
