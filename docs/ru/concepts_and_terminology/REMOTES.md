# Удалённые окружения

Удалённое окружение coast запускает ваши сервисы на удалённой машине вместо вашего ноутбука. Опыт использования CLI и UI идентичен локальным окружениям coast — `coast run`, `coast assign`, `coast exec`, `coast ps` и `coast checkout` работают одинаково. Демон определяет, что экземпляр является удалённым, и направляет операции через SSH-туннель к `coast-service` на удалённом хосте.

## Локальное vs Удалённое

| | Локальный Coast | Удалённый Coast |
|---|---|---|
| DinD container | Запускается на вашей машине | Запускается на удалённой машине |
| Compose services | Внутри локального DinD | Внутри удалённого DinD |
| File editing | Прямое bind-монтирование | Shell coast (локально) + синхронизация rsync/mutagen |
| Port access | Форвардер `socat` | SSH `-L` туннель + форвардер `socat` |
| Shared services | Сеть bridge | Обратный туннель SSH `-R` |
| Build architecture | Архитектура вашей машины | Архитектура удалённой машины |

## Как это работает

Каждое удалённое окружение coast создаёт два контейнера:

1. **Shell coast** на вашей локальной машине. Это лёгкий Docker-контейнер (`sleep infinity`) с теми же bind-монтированиями, что и у обычного coast (`/host-project`, `/workspace`). Он существует для того, чтобы host-агенты могли редактировать файлы, которые синхронизируются на удалённую машину.

2. **Удалённый coast** на удалённой машине, управляемый `coast-service`. Он запускает фактический DinD-контейнер с вашими compose-сервисами, используя динамические порты.

Демон соединяет их через SSH-туннели:

- **Прямые туннели** (`ssh -L`): сопоставляют каждый локальный динамический порт с соответствующим удалённым динамическим портом, чтобы `localhost:{dynamic}` обращался к удалённому сервису.
- **Обратные туннели** (`ssh -R`): предоставляют локальные [общие сервисы](SHARED_SERVICES.md) (Postgres, Redis) удалённому DinD-контейнеру.

## Регистрация удалённых окружений

Удалённые окружения регистрируются в демоне и сохраняются в `state.db`:

```bash
coast remote add my-vm ubuntu@10.0.0.1 --key ~/.ssh/coast_key
coast remote test my-vm
coast remote ls
coast remote rm my-vm
```

Данные подключения (хост, пользователь, порт, SSH-ключ) хранятся в базе данных демона, а не в вашем Coastfile. В Coastfile через секцию `[remote]` задаются только параметры синхронизации.

## Сборки на удалённой машине

Сборки выполняются на удалённой машине, поэтому образы используют её нативную архитектуру. ARM Mac может собирать образы x86_64 на удалённой машине x86_64 без кросс-компиляции.

После сборки артефакт передаётся обратно на вашу локальную машину для повторного использования. Если у другой удалённой машины та же архитектура, предварительно собранный артефакт можно развернуть напрямую без повторной сборки. Подробнее о структуре артефактов сборки см. в [Builds](BUILDS.md).

## Синхронизация файлов

Удалённые окружения coast используют rsync для начальной массовой передачи и mutagen для непрерывной синхронизации в реальном времени. Оба инструмента запускаются внутри контейнеров coast (shell coast и образ `coast-service`), а не на вашей хост-машине. Подробности о настройке синхронизации см. в руководстве [Remote Coasts](../remote_coasts/README.md).

## Управление диском

На удалённых машинах накапливаются Docker volumes, директории рабочих пространств и tarball-архивы образов. Когда `coast rm` удаляет удалённый экземпляр, все связанные ресурсы очищаются. Для осиротевших ресурсов после неудачных операций используйте `coast remote prune`.

## Настройка

Полные инструкции по настройке, включая требования к хосту, развёртывание coast-service и конфигурацию Coastfile, см. в руководстве [Remote Coasts](../remote_coasts/README.md).
