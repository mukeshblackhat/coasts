# Общие сервисы

Общие сервисы — это контейнеры баз данных и инфраструктуры (Postgres, Redis, MongoDB и т. д.), которые запускаются в Docker daemon на вашем хосте, а не внутри Coast. Экземпляры Coast подключаются к ним через bridge-сеть, поэтому каждый Coast обращается к одному и тому же сервису на одном и том же томе хоста.

![Shared services in Coastguard](../../assets/coastguard-shared-services.png)
*Вкладка общих сервисов в Coastguard, показывающая управляемые хостом Postgres, Redis и MongoDB.*

## Как это работает

Когда вы объявляете общий сервис в вашем Coastfile, Coast запускает его в daemon хоста и удаляет его из compose-стека, который выполняется внутри каждого контейнера Coast. Затем Coasts настраиваются так, чтобы направлять трафик по имени сервиса обратно к общему контейнеру, сохраняя при этом порт сервиса со стороны контейнера внутри Coast.

```text
Host Docker daemon
  |
  +--> postgres (host volume: infra_postgres_data)
  +--> redis    (host volume: infra_redis_data)
  +--> mongodb  (host volume: infra_mongodb_data)
  |
  +--> Coast: dev-1  --bridge network--> host postgres, redis, mongodb
  +--> Coast: dev-2  --bridge network--> host postgres, redis, mongodb
```

Поскольку общие сервисы повторно используют ваши существующие тома хоста, любые данные, которые у вас уже есть после локального запуска `docker-compose up`, сразу становятся доступны вашим Coast.

Это различие важно, когда вы используете сопоставленные порты:

```toml
[shared_services.postgis]
image = "ghcr.io/baosystems/postgis:12-3.3"
ports = ["5433:5432"]
```

- На хосте общий сервис публикуется на `localhost:5433`.
- Внутри каждого Coast контейнеры приложения по-прежнему подключаются к `postgis:5432`.
- Простое целое число, например `5432`, является сокращением для тождественного сопоставления `"5432:5432"`.

## Когда использовать общие сервисы

- У вашего проекта есть интеграции MCP, которые подключаются к локальной базе данных — общие сервисы позволяют им продолжать работать без динамического определения портов. Если вы публикуете общий сервис на том же порту хоста, который уже используют ваши инструменты (например, `ports = [5432]`), эти инструменты продолжают работать без изменений. Если вы публикуете его на другом порту хоста (например, `"5433:5432"`), инструменты на стороне хоста должны использовать этот порт хоста, тогда как Coasts продолжают использовать порт контейнера.
- Вам нужны более легковесные экземпляры Coast, поскольку им не нужно запускать собственные контейнеры баз данных.
- Вам не нужна изоляция данных между экземплярами Coast (каждый экземпляр видит одни и те же данные).
- Вы запускаете coding agents на хосте (см. [Filesystem](FILESYSTEM.md)) и хотите, чтобы они получали доступ к состоянию базы данных без маршрутизации через [`coast exec`](EXEC_AND_DOCKER.md). С общими сервисами существующие инструменты базы данных и MCP у агента работают без изменений.

См. страницу [Volume Topology](VOLUMES.md), чтобы узнать об альтернативах, когда изоляция всё же нужна.

## Предупреждение о неоднозначности томов

Имена томов Docker не всегда глобально уникальны. Если вы запускаете `docker-compose up` из нескольких разных проектов, тома хоста, которые Coast подключает к общим сервисам, могут оказаться не теми, которые вы ожидаете.

Перед запуском Coast с общими сервисами убедитесь, что последний `docker-compose up`, который вы выполняли, был из проекта, который вы собираетесь использовать с Coast. Это гарантирует, что тома хоста соответствуют ожиданиям вашего Coastfile.

## Устранение неполадок

Если кажется, что ваши общие сервисы указывают на неправильный том хоста:

1. Откройте интерфейс [Coastguard](COASTGUARD.md) (`coast ui`).
2. Перейдите на вкладку **Shared Services**.
3. Выберите затронутые сервисы и нажмите **Remove**.
4. Нажмите **Refresh Shared Services**, чтобы пересоздать их на основе текущей конфигурации Coastfile.

Это остановит и заново создаст контейнеры общих сервисов, повторно подключив их к правильным томам хоста.

## Общие сервисы и удалённые Coast

При запуске [удалённых coast](REMOTES.md) общие сервисы по-прежнему работают на вашей локальной машине. Daemon устанавливает обратные SSH-туннели (`ssh -R`), чтобы удалённые контейнеры DinD могли обращаться к ним через `host.docker.internal`. Это позволяет удалённым экземплярам совместно использовать вашу локальную базу данных. Для корректной привязки обратных туннелей в sshd на удалённом хосте должен быть включён параметр `GatewayPorts clientspecified`.
