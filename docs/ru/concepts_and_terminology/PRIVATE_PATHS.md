# Приватные пути

Когда несколько экземпляров Coast используют один и тот же корень проекта, они используют одни и те же файлы — и одни и те же inode. Обычно в этом и смысл: изменения файлов на хосте мгновенно появляются внутри Coast, потому что обе стороны видят одну и ту же файловую систему. Но некоторые инструменты записывают состояние, специфичное для процесса, в рабочее пространство и предполагают эксклюзивный доступ, и это предположение нарушается, когда два экземпляра используют одно и то же монтирование.

## Проблема

Рассмотрим Next.js 16, который получает эксклюзивную блокировку на `.next/dev/lock` через `flock(fd, LOCK_EX)` при запуске dev-сервера. `flock` — это механизм ядра на уровне inode — ему не важны пространства имён монтирования, границы контейнеров или пути bind mount. Если два процесса в двух разных контейнерах Coast оба указывают на один и тот же inode `.next/dev/lock` (потому что они используют один и тот же bind mount хоста), второй процесс видит блокировку первого и отказывается запускаться:

```text
⨯ Another next dev server is already running.

- Local: http://localhost:3000
- PID: 1361
- Dir: /workspace/frontend
```

Та же категория конфликтов относится к:

- рекомендательным блокировкам `flock` / `fcntl` (Next.js, Turbopack, Cargo, Gradle)
- PID-файлам (многие демоны записывают PID-файл и проверяют его при запуске)
- кешам сборки, которые предполагают доступ с единственным писателем (Webpack, Vite, esbuild)

Изоляция пространства имён монтирования (`unshare`) здесь не помогает. Пространства имён монтирования управляют тем, какие точки монтирования может видеть процесс, но `flock` работает с самим inode. Два процесса, видящие один и тот же inode через разные пути монтирования, всё равно будут конфликтовать.

## Решение

Поле Coastfile `private_paths` объявляет директории относительно рабочего пространства, которые должны быть отдельными для каждого экземпляра. Каждый экземпляр Coast получает собственный изолированный bind mount для этих путей, основанный на директории для конкретного экземпляра в собственной файловой системе контейнера.

```toml
[coast]
name = "my-app"
private_paths = ["frontend/.next"]
```

После того как Coast монтирует `/workspace` с shared propagation, он применяет дополнительный bind mount для каждого приватного пути:

```text
mkdir -p /coast-private/frontend/.next /workspace/frontend/.next
mount --bind /coast-private/frontend/.next /workspace/frontend/.next
```

`/coast-private/` находится на writable layer контейнера DinD — не на общем bind mount хоста — поэтому каждый экземпляр естественным образом получает отдельные inode. Файл блокировки в `dev-1` находится в другом inode, чем файл блокировки в `dev-2`, и конфликт исчезает.

## Как это работает

Монтирования приватных путей применяются в каждой точке жизненного цикла Coast, где `/workspace` монтируется или перемонтируется:

1. **`coast run`** — после начального `mount --bind /host-project /workspace && mount --make-rshared /workspace` монтируются приватные пути.
2. **`coast start`** — после повторного применения bind mount рабочего пространства при перезапуске контейнера.
3. **`coast assign`** — после размонтирования и повторного монтирования `/workspace` на директорию worktree.
4. **`coast unassign`** — после возврата `/workspace` обратно к корню проекта.

Приватные директории сохраняются между циклами stop/start (они находятся в файловой системе контейнера, а не на общем монтировании). При `coast assign` или `coast unassign` приватные директории **очищаются**, чтобы dev-серверы перекомпилировали файлы из исходников правильной ветки, а не отдавали устаревший результат сборки из предыдущей ветки. При `coast rm` они уничтожаются вместе с контейнером.

## Когда это использовать

Используйте `private_paths`, когда инструмент записывает состояние, специфичное для процесса или экземпляра, в директорию рабочего пространства, и это вызывает конфликты между одновременно работающими экземплярами Coast:

- **Блокировки файлов**: `.next/dev/lock`, `target/.cargo-lock` у Cargo, `.gradle/lock` у Gradle
- **Кеши сборки**: `.next`, `.turbo`, `target/`, `.vite`
- **PID-файлы**: любой демон, который записывает PID-файл в рабочее пространство

Не используйте `private_paths` для данных, которые должны быть общими для экземпляров или видимыми на хосте. Если вам нужны постоянные изолированные данные под управлением Docker (например, тома базы данных), используйте [volumes with `strategy = "isolated"`](../coastfiles/VOLUMES.md).

## Правила валидации

- Пути должны быть относительными (без начального `/`)
- Пути не должны содержать компоненты `..`
- Пути не должны пересекаться — указание одновременно `frontend/.next` и `frontend/.next/cache` является ошибкой, потому что первое монтирование затенит второе

## Связь с Volumes

`private_paths` и `[volumes]` решают разные задачи изоляции:

| | `private_paths` | `[volumes]` |
|---|---|---|
| **Что** | Директории относительно рабочего пространства | Именованные тома под управлением Docker |
| **Где** | Внутри `/workspace` | Произвольные пути монтирования в контейнере |
| **Основаны на** | Локальной файловой системе контейнера (`/coast-private/`) | Именованных томах Docker |
| **Изоляция** | Всегда по экземплярам | Стратегия `isolated` или `shared` |
| **Переживает `coast rm`** | Нет | Isolated: нет. Shared: да. |
| **Сценарий использования** | Артефакты сборки, файлы блокировок, кеши | Базы данных, постоянные данные приложения |
