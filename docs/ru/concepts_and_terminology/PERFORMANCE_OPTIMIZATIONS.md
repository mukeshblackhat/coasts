# Оптимизации производительности

Coast спроектирован так, чтобы переключение веток было быстрым, но в больших монорепозиториях поведение по умолчанию может добавлять лишнюю задержку. На этой странице описаны рычаги, доступные в вашем Coastfile, чтобы уменьшить время assign и unassign.

## Почему Assign может быть медленным

`coast assign` делает несколько вещей при переключении Coast на новый worktree:

```text
coast assign dev-1 --worktree feature/payments

  1. stop affected compose services
  2. create git worktree (if new)
  3. sync gitignored files into worktree (rsync)  ← often the bottleneck
  4. remount /workspace
  5. git ls-files diff  ← can be slow in large repos
  6. restart/rebuild services
```

Две операции дают основную задержку: **синхронизация gitignored-файлов** и **diff через `git ls-files`**. Обе масштабируются с размером репозитория и усиливаются накладными расходами VirtioFS в macOS.

### Синхронизация gitignored-файлов

Когда worktree создаётся впервые, Coast использует `rsync --link-dest`, чтобы через hardlink перенести gitignored-файлы (артефакты сборки, кэши, сгенерированный код) из корня проекта в новый worktree. Hardlink почти мгновенный на файл, но rsync всё равно должен обойти каждую директорию в исходном дереве, чтобы понять, что нужно синхронизировать.

Если корень проекта содержит большие директории, к которым rsync не должен прикасаться — другие worktree, вендоренные зависимости, несвязанные приложения — rsync тратит время, спускаясь в них и делая stat для тысяч файлов, которые он никогда не скопирует. В репозитории с 400 000+ gitignored-файлами один только этот обход может занимать 30–60 секунд.

Coast автоматически исключает `node_modules`, `.git`, `dist`, `target`, `.worktrees`, `.coasts` и другие распространённые тяжёлые директории из этой синхронизации. Дополнительные директории можно исключить через `exclude_paths` в вашем Coastfile (см. ниже).

После того как worktree синхронизирован, записывается маркер `.coast-synced`, и последующие assign для того же worktree полностью пропускают синхронизацию.

### Diff через `git ls-files`

Каждый assign и unassign также запускает `git ls-files`, чтобы определить, какие отслеживаемые файлы изменились между ветками. В macOS весь файловый I/O между хостом и Docker VM проходит через VirtioFS (или gRPC-FUSE на более старых конфигурациях). Операция `git ls-files` делает stat для каждого отслеживаемого файла, и накладные расходы на файл быстро накапливаются. Репозиторий с 30 000 отслеживаемых файлов будет заметно медленнее, чем с 5 000, даже если фактический diff небольшой.

## `exclude_paths` — основной рычаг

Опция `exclude_paths` в вашем Coastfile говорит Coast пропускать целые деревья директорий как во время **синхронизации gitignored-файлов** (rsync), так и во время **diff через `git ls-files`**. Файлы под исключёнными путями всё равно присутствуют в worktree — они просто не обходятся при assign.

```toml
[assign]
default = "none"
exclude_paths = [
    "docs",
    "scripts",
    "test-fixtures",
    "apps/mobile",
]
```

Это самая существенная оптимизация для больших монорепозиториев. Она уменьшает и обход rsync при первом assign, и файловый diff при каждом assign. Если в вашем проекте 30 000 отслеживаемых файлов, но только 20 000 относятся к сервисам, запущенным в Coast, исключение остальных 10 000 убирает треть работы при каждом assign.

### Что исключать

Цель — исключить всё, что вашим сервисам Coast не нужно. Начните с профилирования содержимого репозитория:

```bash
git ls-files | cut -d'/' -f1 | sort | uniq -c | sort -rn
```

Команда показывает количество файлов по директориям верхнего уровня. Далее определите, какие директории ваши compose-сервисы реально монтируют или от каких зависят, и исключите остальные.

**Оставляйте** директории, которые:
- Содержат исходный код, монтируемый в работающие сервисы (например, директории ваших приложений)
- Содержат общие библиотеки, импортируемые этими сервисами
- Упоминаются в `[assign.rebuild_triggers]`

**Исключайте** директории, которые:
- Относятся к приложениям или сервисам, не запущенным в вашем Coast (приложения других команд, мобильные клиенты, CLI-инструменты)
- Содержат документацию, скрипты, CI-конфиги или туллинг, не относящийся к runtime
- Являются большими кэшами зависимостей, закоммиченными в репозиторий (например, вендоренные proto-определения, офлайн-кэш `.yarn`)

### Пример: монорепозиторий с несколькими приложениями

Монорепозиторий с 29 000 файлов в множестве приложений, но релевантны только два:

```text
  13,000  bookface/         ← active
   7,000  ycinternal/       ← active
     850  shared/           ← used by both
   3,800  .yarn/            ← excludable
   2,500  startupschool/    ← excludable
     500  misc/             ← excludable
     300  ycapp/            ← excludable
     ...  (12 more dirs)    ← excludable
```

```toml
[assign]
default = "none"
exclude_paths = [
    ".yarn",
    "startupschool",
    "misc",
    "ycapp",
    "apply",
    "cli",
    "deploy",
    "lambdas",
    # ... any other directories not needed by active services
]
```

Это уменьшает поверхность diff с 29 000 файлов до ~21 000 — примерно на 28% меньше stat-операций при каждом assign.

## Уберите неактивные сервисы из `[assign.services]`

Если ваш `COMPOSE_PROFILES` запускает только подмножество сервисов, уберите неактивные сервисы из `[assign.services]`. Coast вычисляет стратегию assign для каждого перечисленного сервиса, и перезапуск или пересборка сервиса, который не запущен, — это пустая работа.

```toml
# Bad — restarts services that aren't running
[assign.services]
web = "restart"
api = "restart"
mobile-api = "restart"   # not in COMPOSE_PROFILES
batch-worker = "restart"  # not in COMPOSE_PROFILES

# Good — only services that are actually running
[assign.services]
web = "restart"
api = "restart"
```

То же относится и к `[assign.rebuild_triggers]` — удалите записи для сервисов, которые не активны.

## Используйте `"hot"` где возможно

Стратегия `"hot"` полностью пропускает перезапуск контейнера. [Перемонтаж файловой системы](FILESYSTEM.md) подменяет код в `/workspace`, и файловый watcher сервиса (Vite, webpack, nodemon, air и т. д.) автоматически подхватывает изменения.

```toml
[assign.services]
web = "hot"        # Vite/webpack dev server with HMR
api = "restart"    # Rails/Go — needs a process restart
```

`"hot"` быстрее, чем `"restart"`, потому что избегает цикла stop/start контейнера. Используйте его для любых сервисов, которые запускают dev-сервер с отслеживанием файлов. Оставляйте `"restart"` для сервисов, которые загружают код при старте и не отслеживают изменения (большинство приложений на Rails, Go и Java).

## Используйте `"rebuild"` с триггерами

Если стратегия сервиса по умолчанию — `"rebuild"`, то при каждом переключении ветки пересобирается Docker-образ — даже если ничего, влияющего на образ, не изменилось. Добавьте `[assign.rebuild_triggers]`, чтобы привязать пересборку к конкретным файлам:

```toml
[assign.services]
worker = "rebuild"

[assign.rebuild_triggers]
worker = ["Dockerfile", "package.json", "package-lock.json"]
```

Если ни один из файлов-триггеров не изменился между ветками, Coast пропускает rebuild и вместо этого откатывается на restart. Это позволяет избежать дорогих сборок образов при обычных изменениях кода.

## Итоги

| Оптимизация | Эффект | Влияет на | Когда использовать |
|---|---|---|---|
| `exclude_paths` | Высокий | rsync + git diff | Всегда, в любом репозитории с директориями, которые вашему Coast не нужны |
| Удаление неактивных сервисов | Средний | перезапуск сервисов | Когда `COMPOSE_PROFILES` ограничивает, какие сервисы запускаются |
| Стратегия `"hot"` | Средний | перезапуск сервисов | Сервисы с файловыми watcher-ами (Vite, webpack, nodemon, air) |
| `rebuild_triggers` | Средний | пересборка образа | Сервисы со стратегией `"rebuild"`, которым она нужна только при изменениях инфраструктуры |

Начните с `exclude_paths`. Это самое простое и самое эффективное изменение, которое вы можете сделать. Оно ускоряет и первый assign (rsync), и каждый последующий assign (git diff).
