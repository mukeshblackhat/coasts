Você está instalando o Coasts neste projeto. Coast (host em contêiner) é uma ferramenta de CLI que executa vários ambientes de desenvolvimento isolados em uma única máquina usando contêineres Docker-in-Docker. Cada ambiente recebe suas próprias portas, volumes e runtime — ideal para fluxos de trabalho paralelos com worktrees.

Seu trabalho: analisar este projeto e gerar um Coastfile (um arquivo TOML chamado "Coastfile" na raiz do projeto).

=== DOCUMENTAÇÃO ===

O Coast tem documentação embutida acessível a partir da CLI. Use-a para entender o esquema completo do Coastfile, estratégias de volume, comportamento de assign e outras opções de configuração antes de gerar um Coastfile.

Navegue pela árvore de docs:

  coast docs

Isso imprime a árvore completa de docs. Comece lendo os arquivos README — eles fornecem índices para ajudar você a encontrar a documentação certa para cada tópico:

  coast docs --path README.md
  coast docs --path coastfiles/README.md
  coast docs --path concepts_and_terminology/README.md

Leia um doc específico:

  coast docs --path coastfiles/PROJECT.md
  coast docs --path coastfiles/VOLUMES.md

Pesquise nos docs (busca semântica — descreva o que você está procurando em linguagem natural):

  coast search-docs "how do volume strategies work"
  coast search-docs "shared postgres across instances"
  coast search-docs "secret injection from environment variables"

Use os docs para tomar decisões informadas sobre a configuração do Coastfile deste projeto. A seção coastfiles/ cobre cada diretiva do Coastfile em detalhe.

=== ESQUEMA DO COASTFILE (referência rápida) ===

[coast] — Obrigatório. Metadados do projeto.

  name          (string, required)  Identificador do projeto usado na nomeação de contêiner/volume.
  compose       (string, optional)  Caminho para docker-compose.yml relativo ao Coastfile.
  runtime       (string, optional)  "dind" (padrão), "sysbox" ou "podman".
  root          (string, optional)  Sobrescrita da raiz do projeto (relativa ou absoluta).
  worktree_dir  (string or array, optional)  Diretório ou diretórios para worktrees do git (padrão: ".worktrees"). Aceita uma única string ou um array de strings. Auto-detectado a partir de worktrees existentes em runtime.

[coast.setup] — Opcional. Personalize o contêiner DinD em si.

  packages    (array of strings)  Pacotes Alpine para instalar (por exemplo ["nodejs", "npm", "git"]).
  run         (array of strings)  Comandos arbitrários para executar durante o setup.

[ports] — Obrigatório (pelo menos um). Mapa de nome lógico para número de porta.
  Essas portas são encaminhadas para o host quando o coast é feito checkout.

  Exemplo:
  [ports]
  web = 3000
  api = 8080
  postgres = 5432

[volumes.*] — Opcional. Configuração por volume.

  strategy        "isolated" (padrão) ou "shared"
  service         Nome do serviço do Compose que é dono deste volume.
  mount           Caminho de montagem dentro do contêiner do serviço.
  snapshot_source (somente isolated) Semeia a partir de um nome de volume existente.

  Exemplo:
  [volumes.postgres_data]
  strategy = "isolated"
  service = "db"
  mount = "/var/lib/postgresql/data"

[secrets.*] — Opcional. Extração e injeção de segredos.

  extractor   "file", "env", "command" ou "macos-keychain"
  inject      "env:VAR_NAME" ou "file:/path/in/container"
  ttl         Expiração opcional (por exemplo "1h", "30m").

  Parâmetros específicos do extractor:
    file:            path = "./path/to/secret"
    env:             var = "HOST_ENV_VAR"
    command:         run = "echo secret-value"
    macos-keychain:  item = "keychain-item-name"

  Exemplo:
  [secrets.db_password]
  extractor = "env"
  var = "DB_PASSWORD"
  inject = "env:DATABASE_PASSWORD"

[inject] — Opcional. Injeção de arquivo/env do host que não é segredo.

  env     Array de nomes de variáveis de ambiente do host para encaminhar.
  files   Array de caminhos de arquivos do host para montar.

  Exemplo:
  [inject]
  env = ["NODE_ENV", "DEBUG"]
  files = ["~/.ssh/id_ed25519", "~/.gitconfig"]

[shared_services.*] — Opcional. Serviços no daemon Docker do host compartilhados entre instâncias.

  image           Imagem Docker.
  ports           Array de números de porta.
  volumes         Array de montagens de volume.
  env             Tabela inline de variáveis de ambiente.
  auto_create_db  (bool) Criar automaticamente um banco por instância.
  inject          Injetar string de conexão nos contêineres do coast.

  Exemplo:
  [shared_services.postgres]
  image = "postgres:16-alpine"
  ports = [5432]
  volumes = ["postgres_data:/var/lib/postgresql/data"]
  env = { POSTGRES_USER = "dev", POSTGRES_PASSWORD = "dev", POSTGRES_DB = "app" }
  auto_create_db = true
  inject = "env:DATABASE_URL"

[assign] — Opcional. Controla o que acontece ao trocar de branch (coast assign).

  default                 "none", "restart" ou "rebuild"
  [assign.services]       Sobrescritas por serviço.
  [assign.rebuild_triggers]  Globs de arquivo por serviço que disparam rebuild.

  Exemplo:
  [assign]
  default = "none"
  [assign.services]
  api = "restart"
  worker = "rebuild"
  [assign.rebuild_triggers]
  worker = ["Dockerfile", "package.json"]

[services.*] — Opcional. Serviços de processo simples (sem docker-compose).

  command   Comando de shell para executar.
  port      Número de porta.
  restart   "on-failure" ou "always".

  Exemplo:
  [services.web]
  command = "node server.js"
  port = 3000
  restart = "on-failure"

=== EXEMPLO: Mínimo (sem compose) ===

[coast]
name = "my-app"
runtime = "dind"

[coast.setup]
packages = ["nodejs", "npm"]

[ports]
app = 3000

=== EXEMPLO: Com docker-compose ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000
postgres = 5432
redis = 6379

[volumes.postgres_data]
strategy = "shared"
service = "db"
mount = "/var/lib/postgresql/data"

[volumes.redis_data]
strategy = "isolated"
service = "cache"
mount = "/data"

[assign]
default = "none"

[assign.services]
app = "rebuild"

=== EXEMPLO: Com secrets ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000

[secrets.api_key]
extractor = "env"
var = "API_KEY"
inject = "env:API_KEY"

[secrets.ssh_key]
extractor = "file"
path = "~/.ssh/id_ed25519"
inject = "file:/run/secrets/ssh_key"

=== PRINCIPAIS TRADEOFFS PARA DISCUTIR COM O USUÁRIO ===

Antes de gerar o Coastfile, pergunte ao usuário sobre quaisquer escolhas de configuração ambíguas. Aqui estão as principais:

Estratégia de banco de dados e infraestrutura — há três opções para serviços como postgres e redis:
  - Volumes isolados (padrão): cada instância do Coast obtém sua própria cópia dos dados dentro do contêiner DinD. As instâncias não podem interferir entre si. Melhor quando você quer estado de banco por branch.
  - Volumes compartilhados: todas as instâncias leem e escrevem o mesmo volume dentro de seus contêineres DinD. Economiza espaço em disco, mas escritas concorrentes de múltiplas instâncias podem corromper dados.
  - Serviços compartilhados: execute o banco no daemon Docker do host em vez de dentro de cada Coast. Todas as instâncias conectam a um único servidor compartilhado. Usa a menor quantidade de memória, suporta auto_create_db para bancos por instância em um único postgres, e os dados sobrevivem à deleção da instância. Melhor para times grandes ou máquinas com pouca memória.
  - Se o projeto tiver um banco de dados, pergunte ao usuário qual abordagem ele quer. Explique os tradeoffs — isolated é a mais segura, shared services é a mais eficiente em memória.

Estratégia de assign — o que acontece ao alternar um Coast entre worktrees:
  - "none": não fazer nada (para serviços como postgres/redis que não mudam entre branches).
  - "restart": reiniciar o contêiner (para serviços interpretados que só precisam de reinício do processo).
  - "rebuild": reconstruir a imagem Docker e reiniciar (para serviços em que a mudança de branch afeta o Dockerfile ou dependências de build).
  - Se o projeto tiver múltiplos serviços, pergunte quais precisam de rebuild vs restart ao trocar de branch.

=== INSTRUÇÕES ===

1. Examine a estrutura deste projeto. Se houver um docker-compose.yml, leia-o para identificar serviços, portas e volumes.
2. Detecte o diretório de worktree do git existente. Execute `git worktree list` para verificar se o projeto já tem worktrees do git configuradas.
   - Se existirem worktrees, examine seus caminhos para determinar o diretório pai comum (por exemplo, se as worktrees estão em `../.worktrees/feat-a` e `../.worktrees/feat-b`, o worktree_dir é `"../.worktrees"`).
   - Defina `worktree_dir` no Coastfile para corresponder ao diretório detectado.
   - Se não existirem worktrees, omita `worktree_dir` (o Coast usa ".worktrees" por padrão). NÃO use ".coasts" — isso polui o projeto com um diretório com marca do Coast.
3. Pergunte ao usuário se ele usa algum destes coding harnesses com este projeto:
   - **Claude Code** — worktrees em `.claude/worktrees`
   - **OpenAI Codex** — worktrees em `~/.codex/worktrees`
   - **Cursor** — worktrees em `~/.cursor/worktrees/<name>` (onde `<name>` é o nome do coast em `[coast] name`)
   - **Conductor** — worktrees em `~/conductor/workspaces/<name>`
   - **T3 Code** — worktrees em `~/.t3/worktrees/<name>`
   Para cada harness que o usuário selecionar, inclua seu diretório de worktree no array `worktree_dir`. Combine-os com qualquer diretório detectado na etapa 2. Se o usuário não selecionar nenhum e nenhuma worktree tiver sido detectada na etapa 2, omita `worktree_dir` (o Coast usa ".worktrees" por padrão).
4. Leia os docs relevantes do Coast (use `coast docs` e `coast search-docs`) para entender estratégias de volume, comportamento de assign e quaisquer opções de configuração que se apliquem à stack deste projeto.
5. Pergunte ao usuário sobre quaisquer escolhas de configuração ambíguas (veja os tradeoffs acima). Não chute — explique as opções e deixe que ele decida.
6. Gere um Coastfile na raiz do projeto com base na análise do projeto e na entrada do usuário.
7. Se o projeto não tiver docker-compose.yml, use [services.*] para definições de processos simples ou [coast.setup] para instalar dependências.
8. Execute `coast build`. Se falhar, verifique o erro e consulte os docs (`coast search-docs "<error description>"`) para solucionar.
9. Execute `coast run dev-1`. Se falhar, verifique o erro e consulte os docs.
10. Execute `coast ui` para abrir o dashboard do Coastguard (isto é para o usuário quando você terminar).
11. **Configure habilidades dos harnesses (Recomendado).** Agora que o Coastfile está configurado e o Coast está em execução, é fortemente recomendado configurar as habilidades do Coast para os harnesses que o usuário selecionou na etapa 3. Essas habilidades ensinam o agente de IA em cada harness a usar Coasts — sem elas, o harness não saberá como executar testes, ler logs ou gerenciar instâncias do Coast.

   Pergunte ao usuário: "Seu Coast está no ar e em execução! Eu recomendaria configurar agora as habilidades do Coast para [lista dos harnesses da etapa 3] — isso leva só um momento e permitirá que suas ferramentas de programação com IA funcionem com Coasts automaticamente. Quer que eu faça isso?"

   Se o usuário concordar, execute:

     coast harness-setup-prompt --harness <name1> --harness <name2> ...

   passando todos os harnesses que o usuário selecionou na etapa 3. Siga as instruções na saída para configurar cada harness. Como você já verificou a CLI na etapa 1 e configurou `worktree_dir` na etapa 3, pule essas etapas nas instruções do harness (a saída lembrará você disso quando houver múltiplos harnesses).

   Se o usuário recusar, informe que ele pode configurar as habilidades mais tarde executando `coast harness-setup-prompt --harness <name>` e seguindo as instruções.
