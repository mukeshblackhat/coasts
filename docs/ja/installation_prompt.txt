あなたはこのプロジェクトに Coasts をインストールしています。Coast（コンテナ化されたホスト）は CLI ツールで、Docker-in-Docker コンテナを使用して、1 台のマシン上で複数の隔離された開発環境を実行します。各環境はそれぞれ独自のポート、ボリューム、ランタイムを持つため、並列の worktree ワークフローに最適です。

あなたの仕事: このプロジェクトを分析し、Coastfile（プロジェクトルートに置く "Coastfile" という名前の TOML ファイル）を生成してください。

=== ドキュメント ===

Coast には CLI からアクセスできる組み込みドキュメントがあります。Coastfile を生成する前に、これらを使って Coastfile スキーマ全体、ボリューム戦略、assign の挙動、その他の設定オプションを理解してください。

ドキュメントツリーを参照:

  coast docs

これにより、完全なドキュメントツリーが出力されます。まず README ファイルを読むところから始めてください — 各トピックに適したドキュメントを見つけるためのインデックスが提供されています:

  coast docs --path README.md
  coast docs --path coastfiles/README.md
  coast docs --path concepts_and_terminology/README.md

特定のドキュメントを読む:

  coast docs --path coastfiles/PROJECT.md
  coast docs --path coastfiles/VOLUMES.md

ドキュメントを検索（セマンティック検索 — 探している内容を自然言語で説明）:

  coast search-docs "how do volume strategies work"
  coast search-docs "shared postgres across instances"
  coast search-docs "secret injection from environment variables"

これらのドキュメントを使って、このプロジェクトの Coastfile 設定について十分な情報に基づいた判断をしてください。coastfiles/ セクションでは、すべての Coastfile ディレクティブが詳細に解説されています。

=== COASTFILE スキーマ（クイックリファレンス） ===

[coast] — 必須。プロジェクトのメタデータ。

  name          (string, required)  コンテナ/ボリューム命名に使われるプロジェクト識別子。
  compose       (string, optional)  Coastfile からの相対パスで指定する docker-compose.yml のパス。
  runtime       (string, optional)  "dind"（デフォルト）、"sysbox"、または "podman"。
  root          (string, optional)  プロジェクトルートの上書き（相対または絶対）。
  worktree_dir  (string or array, optional)  git worktree 用のディレクトリまたはディレクトリ群（デフォルト: ".worktrees"）。単一の文字列または文字列の配列を受け付けます。実行時に既存の worktree から自動検出されます。

[coast.setup] — 任意。DinD コンテナ自体をカスタマイズ。

  packages    (array of strings)  インストールする Alpine パッケージ（例: ["nodejs", "npm", "git"]）。
  run         (array of strings)  セットアップ中に実行する任意のコマンド。

[ports] — 必須（少なくとも 1 つ）。論理名からポート番号へのマップ。
  これらのポートは coast がチェックアウトされたときにホストへフォワードされます。

  例:
  [ports]
  web = 3000
  api = 8080
  postgres = 5432

[volumes.*] — 任意。ボリュームごとの設定。

  strategy        "isolated"（デフォルト）または "shared"
  service         このボリュームを所有する Compose サービス名。
  mount           サービスコンテナ内のマウントパス。
  snapshot_source (isolated のみ) 既存のボリューム名からシード。

  例:
  [volumes.postgres_data]
  strategy = "isolated"
  service = "db"
  mount = "/var/lib/postgresql/data"

[secrets.*] — 任意。シークレットの抽出と注入。

  extractor   "file"、"env"、"command"、または "macos-keychain"
  inject      "env:VAR_NAME" または "file:/path/in/container"
  ttl         任意の有効期限（例: "1h"、"30m"）。

  extractor 固有パラメータ:
    file:            path = "./path/to/secret"
    env:             var = "HOST_ENV_VAR"
    command:         run = "echo secret-value"
    macos-keychain:  item = "keychain-item-name"

  例:
  [secrets.db_password]
  extractor = "env"
  var = "DB_PASSWORD"
  inject = "env:DATABASE_PASSWORD"

[inject] — 任意。非シークレットのホストファイル/環境変数の注入。

  env     フォワードするホスト環境変数名の配列。
  files   マウントするホストファイルパスの配列。

  例:
  [inject]
  env = ["NODE_ENV", "DEBUG"]
  files = ["~/.ssh/id_ed25519", "~/.gitconfig"]

[shared_services.*] — 任意。ホストの Docker デーモン上で動作し、インスタンス間で共有されるサービス。

  image           Docker イメージ。
  ports           ポート番号の配列。
  volumes         ボリュームマウントの配列。
  env             環境変数のインラインテーブル。
  auto_create_db  (bool) インスタンスごとのデータベースを自動的に作成。
  inject          接続文字列を coast コンテナへ注入。

  例:
  [shared_services.postgres]
  image = "postgres:16-alpine"
  ports = [5432]
  volumes = ["postgres_data:/var/lib/postgresql/data"]
  env = { POSTGRES_USER = "dev", POSTGRES_PASSWORD = "dev", POSTGRES_DB = "app" }
  auto_create_db = true
  inject = "env:DATABASE_URL"

[assign] — 任意。ブランチ切り替え時（coast assign）の挙動を制御。

  default                 "none"、"restart"、または "rebuild"
  [assign.services]       サービスごとの上書き。
  [assign.rebuild_triggers]  再ビルドをトリガーするサービスごとのファイル glob。

  例:
  [assign]
  default = "none"
  [assign.services]
  api = "restart"
  worker = "rebuild"
  [assign.rebuild_triggers]
  worker = ["Dockerfile", "package.json"]

[services.*] — 任意。プロセスのみのサービス（docker-compose 不要）。

  command   実行するシェルコマンド。
  port      ポート番号。
  restart   "on-failure" または "always"。

  例:
  [services.web]
  command = "node server.js"
  port = 3000
  restart = "on-failure"

=== 例: 最小（compose なし） ===

[coast]
name = "my-app"
runtime = "dind"

[coast.setup]
packages = ["nodejs", "npm"]

[ports]
app = 3000

=== 例: docker-compose あり ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000
postgres = 5432
redis = 6379

[volumes.postgres_data]
strategy = "shared"
service = "db"
mount = "/var/lib/postgresql/data"

[volumes.redis_data]
strategy = "isolated"
service = "cache"
mount = "/data"

[assign]
default = "none"

[assign.services]
app = "rebuild"

=== 例: secrets あり ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000

[secrets.api_key]
extractor = "env"
var = "API_KEY"
inject = "env:API_KEY"

[secrets.ssh_key]
extractor = "file"
path = "~/.ssh/id_ed25519"
inject = "file:/run/secrets/ssh_key"

=== ユーザーと議論すべき主要なトレードオフ ===

Coastfile を生成する前に、曖昧な設定の選択肢についてユーザーに確認してください。主なものは次のとおりです:

データベースおよびインフラ戦略 — postgres や redis のようなサービスには 3 つの選択肢があります:
  - 隔離ボリューム（デフォルト）: 各 Coast インスタンスは DinD コンテナ内にデータのコピーをそれぞれ持ちます。インスタンス同士が干渉できません。ブランチごとの DB 状態が欲しい場合に最適です。
  - 共有ボリューム: すべてのインスタンスが DinD コンテナ内の同じボリュームを読み書きします。ディスク容量は節約できますが、複数インスタンスからの同時書き込みでデータが破損する可能性があります。
  - 共有サービス: 各 Coast 内ではなく、ホストの Docker デーモン上でデータベースを実行します。すべてのインスタンスは 1 つの共有サーバーに接続します。メモリ使用量が最小で、単一の postgres 上でインスタンスごとの DB を作る auto_create_db をサポートし、データはインスタンス削除後も残ります。大規模チームやメモリ制約のあるマシンに最適です。
  - プロジェクトにデータベースがある場合、どのアプローチにしたいかユーザーに確認してください。トレードオフを説明してください — 隔離が最も安全で、共有サービスが最もメモリ効率が良いです。

Assign 戦略 — Coast を worktree 間で切り替えたときに何が起きるか:
  - "none": 何もしない（ブランチ間で変化しない postgres/redis のようなサービス向け）。
  - "restart": コンテナを再起動（プロセス再起動だけでよいインタプリタ系サービス向け）。
  - "rebuild": Docker イメージを再ビルドして再起動（ブランチ変更が Dockerfile やビルド依存に影響するサービス向け）。
  - プロジェクトに複数サービスがある場合、ブランチ切り替え時に再ビルドが必要なものと再起動でよいものをユーザーに確認してください。

=== 手順 ===

1. このプロジェクトの構成を確認します。docker-compose.yml がある場合はそれを読み、サービス、ポート、ボリュームを特定します。
2. 既存の git worktree ディレクトリを検出します。`git worktree list` を実行し、プロジェクトがすでに git worktree を設定しているか確認します。
   - worktree が存在する場合、パスを調べて共通の親ディレクトリを特定します（例: worktree が `../.worktrees/feat-a` と `../.worktrees/feat-b` にあるなら、worktree_dir は `"../.worktrees"`）。
   - 検出したディレクトリに合わせて Coastfile の `worktree_dir` を設定します。
   - worktree が存在しない場合、`worktree_dir` は省略します（Coast のデフォルトは ".worktrees"）。".coasts" は使用しないでください — Coast ブランドのディレクトリでプロジェクトを汚染します。
3. このプロジェクトで次のいずれかのコーディングハーネスを使っているか、ユーザーに確認します:
   - **Claude Code** — worktree は `.claude/worktrees`
   - **OpenAI Codex** — worktree は `~/.codex/worktrees`
   - **Cursor** — worktree は `~/.cursor/worktrees/<name>`（`<name>` は `[coast] name` の coast 名）
   - **Conductor** — worktree は `~/conductor/workspaces/<name>`
   - **T3 Code** — worktree は `~/.t3/worktrees/<name>`
   ユーザーが選んだ各ハーネスについて、その worktree ディレクトリを `worktree_dir` 配列に含めます。これをステップ 2 で検出したディレクトリと組み合わせます。ユーザーがどれも選ばず、かつステップ 2 で worktree が検出されなかった場合は、`worktree_dir` を省略します（Coast のデフォルトは ".worktrees"）。
4. 関連する Coast ドキュメント（`coast docs` と `coast search-docs` を使用）を読み、ボリューム戦略、assign の挙動、このプロジェクトのスタックに適用される設定オプションを理解します。
5. 曖昧な設定の選択肢についてユーザーに確認します（上記のトレードオフ参照）。推測しないでください — 選択肢を説明し、ユーザーに決めてもらいます。
6. プロジェクト分析とユーザー入力に基づいて、プロジェクトルートに Coastfile を生成します。
7. プロジェクトに docker-compose.yml がない場合は、素のプロセス定義には [services.*] を使うか、依存関係のインストールに [coast.setup] を使用します。
8. `coast build` を実行します。失敗した場合はエラーを確認し、ドキュメント（`coast search-docs "<error description>"`）を参照してトラブルシュートします。
9. `coast run dev-1` を実行します。失敗した場合はエラーを確認し、ドキュメントを参照します。
10. `coast ui` を実行して Coastguard ダッシュボードを開きます（これは作業完了時にユーザーが使用するものです）。
11. **ハーネススキルをセットアップする（推奨）**。Coastfile の設定が完了し、Coast が実行中になったので、ステップ 3 でユーザーが選択したハーネス向けに Coast スキルをセットアップすることを強く推奨します。これらのスキルは、各ハーネス内の AI エージェントに Coasts の使い方を教えます。これがないと、ハーネスはテストの実行、ログの確認、または Coast インスタンスの管理方法を認識できません。

   次のようにユーザーに尋ねてください: "Your Coast is up and running! I'd recommend setting up the Coast skills for [list the harnesses from step 3] now — this only takes a moment and will let your AI coding tools work with Coasts automatically. Want me to do that?"

   ユーザーが同意した場合は、次を実行します:

     coast harness-setup-prompt --harness <name1> --harness <name2> ...

   ステップ 3 でユーザーが選択したすべてのハーネスを渡してください。出力内の手順に従って、各ハーネスをセットアップします。ステップ 1 で CLI の確認はすでに済んでおり、ステップ 3 で `worktree_dir` も設定済みなので、ハーネス手順内のそれらのステップはスキップしてください（複数のハーネスがある場合、出力でもその旨が案内されます）。

   ユーザーが辞退した場合は、後で `coast harness-setup-prompt --harness <name>` を実行し、手順に従うことでスキルをセットアップできることを伝えてください。
